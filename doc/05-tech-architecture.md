# 05. 기술 아키텍처

> 웹 기반, 모바일 우선 → PC 확장. 1인 개발에 맞춘 가볍고 검증된 스택을 우선한다.
> 가장 큰 기술 도전은 **전략↔전술의 연속 줌**과 **이중 시계(전략 빠름/전술 느림)**,
> 그리고 둘을 잇는 **이벤트 버스·상태 인계**다. (Hex 전장 기반)

## 5.1 기술 원칙

1. **웹 우선, 한 코드베이스** — 모바일 브라우저와 PC 브라우저에서 같은 코드가 돌게.
2. **모바일 성능이 기준선** — 모바일에서 안정적 30~60fps. PC는 자연히 여유.
3. **1인이 감당할 스택** — 익숙하고 생태계가 큰 도구. 새로움보다 생산성.
4. **나중에 네이티브로** — 처음엔 웹, 필요 시 Capacitor/PWA로 앱 스토어 배포.

## 5.2 스택 후보 (논의·결정 필요)

### 렌더링 / 게임 엔진
2D 탑다운 + 연속 줌이 목표. 무거운 3D 엔진은 불필요하되, **줌 연출**은 잘 받쳐줘야 한다.

| 후보 | 장점 | 단점 | 적합도 |
|------|------|------|--------|
| **PixiJS** (+ 직접 게임 로직) | 빠른 2D 렌더, 가벼움, 카메라 줌/스케일 자유로움 | 게임 구조는 직접 설계 | ★ 권장 |
| **Phaser** | 게임 전용 기능 내장(씬/입력/카메라) | Pixi보다 무거움 | 대안 |
| Godot (웹 export) | 풀 게임 엔진, 에디터 | 웹 빌드 용량·모바일 성능 이슈 | 보류 |

> **권장 출발점:** 렌더링은 **PixiJS**(카메라 스케일로 연속 줌 구현), 게임 로직은 TypeScript로 직접.
> UI 패널(경영 내정/메뉴)은 가벼운 프레임워크 또는 순수 DOM. 게임 루프와 UI 상태는 분리.

### 언어 / 빌드
- **TypeScript** (타입 안정성 → 1인 개발의 버그 방어선).
- **Vite** (빠른 개발 서버·번들).

### 상태 / 데이터
- 게임 상태는 **순수 데이터 객체 + 결정적(deterministic) 시뮬레이션**.
  → 자동 해결 전투, 세이브/로드, 향후 리플레이/멀티에 모두 유리.
  - **운(運)=시드 RNG**: 무작위 판정은 **전역 시드 고정** 의사난수만 사용 → 같은 시드·입력이면 100% 재현.
    RNG를 "시드 의존 결정함수"로 취급해 결정성과 양립시킨다. (→ [07 7.1](./07-entities-data.md))
- 저장: 로컬 IndexedDB/localStorage. 클라우드 세이브는 후순위.

### 백엔드
- 프로토타입은 **백엔드 없음(클라이언트 온리)**. 정적 호스팅으로 배포.
- 향후 비동기 멀티/랭킹이 필요하면 경량 백엔드(Supabase/Firebase) 추가.

```
│                        UI 레이어                          │  DOM/경량 프레임워크
│        경영 내정 패널 · 메뉴 · HUD · 명령 버튼            │
├──────────────────────────────────────────────────────────┤
│                  렌더 레이어 (PixiJS)                     │
│        카메라 스케일 = 연속 줌 (전략 ↔ 전술 한 화면)      │
├──────────────────────────────────────────────────────────┤
│                게임 시뮬레이션 (TS, 순수)                 │
│  ┌────────────┐   전투   ┌──────────────────────────────┐ │
│  │ 경영 시뮬   │ ──개시─→ │   전투 시뮬 (공유 실시간)     │ │
│  │ (턴제/맵)   │ ←─결과── │  ┌────────────┐ 줌 ┌────────┐│ │
│  └────────────┘          │  │ 전략(부대)  │←─→│전술(병종)││ │
│                          │  └────────────┘   └────────┘│ │
│                          └──────────────────────────────┘ │
├──────────────────────────────────────────────────────────┤
│   시간 관리자: 이중 시계 (전략 빠름 / 전술 느림) +        │  ★ TIME_RATIO
│   Active Pause(둘 동시 정지)  +  이벤트 버스(전략→전술)    │
├──────────────────────────────────────────────────────────┤
│                저장/로드 (IndexedDB)                      │
└──────────────────────────────────────────────────────────┘
```

> **핵심 모듈 셋:**
> - **시간 관리자** — 전략·전술 **두 시계**를 `TIME_RATIO`로 함께 굴린다(전략이 빠름). 액티브 포즈는
>   양쪽을 동시에 멈춘다.
> - **전투 시뮬** — Hex 전장 위 부대 단위 자동 계산이 기본. **줌인한 교전 하나만** 개별 병사 해상도로 정밀 시뮬.
> - **이벤트 버스** — 전략에서 일어난 사건(부대 후퇴·증원 접근 등)을 전술 화면 **알림**으로 전달. → [04 문서 4.4](./04-tactics-layer.md)

## 5.4 이중 시계 · 줌 · 상태 인계 구현 노트

- **줌 & 좌표:** PixiJS 카메라(stage scale/position)로 구현.
  - 멀리(전략): 부대 = Hex 좌표 토큰/카운터(+전방 화살표).
  - 가까이(전술): 병종 그룹을 **연속 좌표 + 대형 덩어리**(앵커 + 대형 슬롯 스프라이트 + 노이즈)로 렌더.
    **개별 병사 독립 시뮬 X** — 그룹 단위 시뮬 + 시각적 스캐터. → [04 4.2.1](./04-tactics-layer.md)
- **이중 시계:** 한 고정 timestep 위에서 전략은 매 틱, 전술은 `1/TIME_RATIO` 비율로 누적 진행.
  → "전략 빠름·전술 느림"을 timestep 배율로 단순 구현. 액티브 포즈는 누적을 멈춘다.
- **시뮬 해상도:** 전술 상세 시뮬은 **줌인한 교전 한 건**만. 나머지 부대/교전은 부대 단위 자동 계산.
  → 모바일 동시 부하가 작다(한 번에 한 상세 전장).
- **레이어 격리 & 최소 인터페이스 (확정 2026-07-01):** 전술은 **독립 레이어**다. 전략 Hex 좌표를
  전술 연속 좌표로 되돌리는 **양방향 위치 변환은 필요 없다.**
  - **진입(전략→전술):** 전략에서 결정된 **교전 스냅샷**(참가 유닛·병종·병력·사기·장수·지형)을 넘겨
    전술의 **초기 상태**로 쓴다. 이후 전술은 별개로 전개.
  - **복귀(전술→전략):** 전술이 전략에 되돌리는 것은 **유닛의 승/패뿐.** (세부 사상자·배치·좌표는 되돌리지 않음)
  - 시간만 `TIME_RATIO`로 공유 + [알림 이벤트 버스](#). 상태 결합은 위 두 줄이 전부.
- **결정성:** 고정 timestep으로 시뮬, 렌더는 보간. 자동 해결과 실시간 결과가 일치하도록.

## 5.5 모바일 우선 고려사항

- **입력:** 터치(탭/드래그/**핀치-줌**)를 1차로. 핀치-줌이 전략↔전술 전환의 핵심 제스처.
- **화면 방향:** 경영맵=세로 친화, 전투(전략/전술)=가로 친화일 수 있음 → 조기 결정 필요.
- **성능:** 화면 내 동시 스프라이트 상한, 오브젝트 풀링, 배칭. 확대 시에만 병사 스프라이트 생성.
- **세션:** 액티브 포즈 + 자동 저장으로 언제든 끊고 이어하기.

## 5.6 전술 슬라이스 구현 설계 (전술 우선 — 즉시 빌드 대상, 2026-07-03)

개발 순서를 **전술 우선**으로 잡았다([06 6.1](./06-prototype-scope.md)). **이 절이 지금 만들 것**이다.
Hex·이중 시계·전략 연동은 v0에 **없다**(2단계 이후). v0 = **격리된 전술 교전 하나**([04 4.8](./04-tactics-layer.md)).

### 5.6.1 원칙 — 헤드리스 결정적 시뮬 + 렌더 분리
- **시뮬 코어는 PixiJS를 모른다.** 순수 TS, **고정 timestep**, **시드 RNG**. → 나중에 자동 해결·테스트·리플레이에 그대로 재사용.
- **렌더는 로직을 모른다.** 매 프레임 시뮬 상태를 읽어 그리고 **보간**만. (액티브 포즈 = 시뮬 정지, 렌더는 계속)
- **밸런싱 수치 = 데이터**(등급→계수·지형·편성 = JSON/TS), 코드에 하드코딩 금지.
- **입력=스냅샷 / 출력=승패+정도** 경계 유지([04 4.6](./04-tactics-layer.md)).
- **비용 모델 (스파이크로 확인 2026-07-06):** 전투 연산 = **덩어리끼리 = O(부대 수)** — 병사(스프라이트) 수와 무관하게 극소.
  렌더 = O(스프라이트)이나 GPU 배칭으로 쌈(**스파이크: 4000 스프라이트 @120fps 데스크탑**).
  ⚠️ **스프라이트에 개별 로직 금지** — 개별 병사 충돌/AI를 넣는 순간 O(병사²)로 폭발. 스프라이트는 끝까지 **순수 렌더**([04 4.2.1](./04-tactics-layer.md)).

### 5.6.2 모듈 구조 (4)
```
data/     병종 등급표·등급→계수·지형 템플릿 3종·교전 스냅샷        (순수 데이터)
sim/      Battle 틱 루프 · 이동/전투/사기/도주 · 시드 RNG          (순수 TS, 헤드리스)
render/   PixiJS: 덩어리 스캐터·facing·핀치줌 카메라·보간          (상태 읽기 전용)
input/    액티브 포즈 · 명령 입력(이동배치/돌격/방어전념) → sim 명령 (UI→sim)
```

**렌더 — 스프라이트 매핑 (2026-07-06):**
- sim이 준 **슬롯(width×depth, facing 회전)** 위치에 스프라이트를 얹기만 한다. 스프라이트는 **독립 이동·AI 없음.**
- 각 스프라이트 = **루프 애니(AnimatedSprite) + 렌더 전용 상하 bob**(sin, 슬롯별 위상차) = "꿈틀대는 덩어리".
- ⚠️ **bob·애니는 순수 렌더 — sim에 절대 되먹이지 않음**(결정성). sim은 슬롯 위치만 안다.
- **병력수↔스프라이트 축약(1:N)** 로 개수 관리 + **텍스처 아틀라스 배칭·풀링**. 모바일 상한은 **0단계 스파이크에서 검증.**
- facing = 슬롯 격자 회전 + 스프라이트 좌우 flip(v0). 방향별 걷기 애니는 후순위.

### 5.6.3 데이터 모델 (초안)
```
Battle   { terrain, unitA, unitB, clock, rng(seed), phase: 배치|교전|도주|종료 }
Unit     { side, 사기(0~100), general, flag, cohorts[] }
Cohort   { 병종, 전사가능HP, 부상HP, anchor{x,y}, facing(rad),
           대형{ width, depth(≥10) }, 상태: 정지|이동|방어전념, 목표 }
General  { pos, hp, 상태: 출진|휴식|대기, 성향:근접 }
Flag     { pos, 명령반경, hp }          # v0 깃발병 = 이 Flag
```
- **병력수 = 전사가능HP + 부상HP.** 피해 → 치명율로 두 통에 분배. 부상HP는 전술 중 회복 없음([03 3.6.2](./03-strategy-layer.md)).
- **대형 width = 병력 ÷ depth**(기본 depth 10). 사상자 → width가 **양끝부터** 줄어듦(중앙 수렴).

### 5.6.4 틱 루프 & 결정성
```
고정 timestep(예 1/30s)마다:
  1) 이동/회전 적용(이동력·회전속도, 덩어리 모임/펼침, 기병 가속·선회)
  2) 전선 접촉 판정 → 접전 폭 산출
  3) 피해 계산(접전 폭 × 공속 × 공/방 ÷ ... × 지형·facing·사기) → 전사/부상 분배
  4) 사기 갱신 → 한쪽 0이면 phase=도주
  5) (도주 phase) 화면 끝 이동 + 피격 누적, 10초/도달 시 phase=종료 → 승패·정도
액티브 포즈: 이 루프를 멈춤. 재개 시 입력된 명령 반영.
RNG: Date.now/Math.random 금지 — 시드 PRNG만(리플레이·디버그 재현).
```

### 5.6.5 핵심 난제 3 (여기가 리스크)
1. **접전 폭 기하** ★가장 중요 — 연속 좌표 덩어리에서 "접촉면 폭"을 어떻게 뽑나.
   - 제안: 각 Cohort를 **전면 선분**(anchor 중심, facing에 수직, 길이=width)으로 근사 → 두 선분이 접촉거리 안일 때 **투영 겹침 길이 = 실효 전면**. 단순·결정적·모델과 정합.
2. **덩어리 이동/표현** — anchor 보간 이동 + 슬롯(width×depth) 배치 + 노이즈 스캐터. 이동 중 width 축소(모임)→도착 시 확대(펼침). 사상자 시 좌우→중앙 수축.
3. **기병** ★가장 복잡 — 가속(2초)→최대속도 charge(공A→S), **돌파**(전면 병목 우회→뒷열 타격), **선회**(반경≈장수 20배). v0에서 가장 늦게, 따로 검증.

#### 접전 폭 기하 — 전면 선분 모델 (상세 2026-07-06)
**렌더 슬롯과 같은 기하**(anchor·facing·width·depth)를 sim이 전투 계산에 그대로 쓴다.
```
Cohort: anchor P, facing θ (d=(cosθ,sinθ) 단위벡터), width W, depth D
전면 선분 = 대형의 앞 모서리
  전면중심 F = P + d·(D/2)              # anchor=대형 중심 기준
  수직 n = (−sinθ, cosθ)
  선분 양끝 = F ± n·(W/2)               # 길이 W
```
**접촉 판정 & 실효 전면(=접전 폭):**
```
1) 두 Cohort 전면 선분 간격 ≤ 접촉거리 AND 서로 마주봄(facing 내적 < 0)
2) 접촉축 = 두 facing 평균의 수직 방향
3) 양측 선분 끝점을 접촉축에 투영 → 구간 [a0,a1], [b0,b1]
4) 실효 전면 = max(0, min(a1,b1) − max(a0,b0))     # 겹침 길이(월드 단위)
5) 기본 공격 횟수 = 실효 전면 ÷ (10·병사간격)        # → 03 3.6.2
```
**엣지 케이스:**
- **사선/측면 접촉:** facing 각도 크면 투영 겹침이 짧아짐 → 자연히 유효 전면↓. **측·후면 피격**은 수비측 방어 감소(방패 측70/후30)로 별도 반영.
- **지형 병목:** 실효 전면을 **통로 폭으로 clamp**(min). → 다리·애로에서 소수 저지.
- **부분 감쌈/포위:** v0 1:1에선 무시. 협공(2:1)은 전면 외 **추가 접촉 구간**으로 확장(v0.5+).
- **한계:** 선분 근사라 곡선 전선·부분 관통은 단순화됨. **C단계에서 실측 후 부족하면 보강**(다중 선분 등).

### 5.6.6 빌드 순서 (스파이크 → 슬라이스)
- **0단계 스파이크:** PixiJS 연속공간 + **한 덩어리** 렌더(앵커+슬롯+노이즈) + facing 회전 + 핀치줌. *전투 없음* — "움직이고 도는 덩어리가 병사를 흘리며 그려지나".
- **1단계 (점진):**
  - **A** sim 코어 뼈대: 데이터 모델·고정 틱·스냅샷 로드(헤드리스, 콘솔 검증).
  - **B** 이동/배치 명령 + 덩어리 이동(모임→펼침) + facing 회전.
  - **C** 전선 접촉 + **접전 폭** + 근접 피해(공속·치명율) + 병력 수축.
  - **D** 궁병 사격(정지·사거리) + 사기 + 종료(사기0)·도주·정도.
  - **E** 장수(근접·명령반경) + **기병**(가속·돌파·선회).
  - **F** 언덕 보정 + 지형 템플릿 3종.
- 각 단계마다 **헤드리스 sim이 콘솔에서 맞는지 먼저**, 그다음 렌더 붙이기.
