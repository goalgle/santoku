# 06. 프로토타입 범위 & 로드맵

> 1인 개발. 핵심은 "재미가 검증되는 가장 작은 빌드"를 빠르게 만드는 것.

## 6.1 프로토타입의 목적 (무엇을 증명하는가)

이 게임의 정체성은 **연속 줌**이다 — 부대를 지휘하다(전략) 한 유닛으로 파고들어(전술)
장수·병종·지형을 다루고, 다시 빠져나오는 흐름. 따라서 증명할 질문:

> **"전략에서 부대를 지휘하다 특정 유닛을 확대해 전술적으로 개입하는 흐름이 재미있는가?"**
> **"그 전환이 매끄럽고, 액티브 포즈로 판단하는 맛이 사는가?"**

전투(전략+전술)를 **먼저** 검증한다. 경영(턴제)은 그 전투에 의미를 줄 만큼만 최소로 붙인다.

## 6.2 MVP 범위 (있는 것 / 없는 것)

### ✅ MVP에 포함
- **전략 전투 1종** (한 개의 **Hex 전장**, 실시간 + 액티브 포즈)
  - Hex 그리드, **한 타일 한 유닛**, 부대 **전방(facing)** + 측후면 보정
  - 병종 3종(창/궁/기) + 상성 + 사기/패주
  - 부대 6~12개, **교전 자동 계산**, 부대 단위 명령(선택/이동/전방/정지·재개)
- **전술 확대 (★ 핵심 검증)**
  - 한 교전을 확대 → 장수·개별 병종 표시, 지형 연출 1~2종(예: 언덕)
  - 전술 명령 2~3종(예: 장수 돌격 / 병종별 지시 / 지형 활용)
  - **이중 시계**(전략 빠름/전술 느림, `TIME_RATIO`) + 줌인 교전 1건만 정밀 시뮬
  - **전략 상황 알림** 1~2종(예: "A부대 후퇴 중") → 주의의 안개 검증
- **최소 경영 레이어**
  - 거점 6~8개 노드 맵, 턴제, 부대 이동 → 접촉 시 전투 발생
  - 자원 2~3종, 도시에서 부대 편성, 자동 해결 옵션
- **AI 1세력** (단순 규칙 기반)
- **세이브/로드 1슬롯**

### ❌ MVP에서 제외 (나중에)
- 전략↔전술 완전 동시 실시간(옵션 A) — 우선 단순안으로
- 발동형 지형 다수(낙석 등) — 1종만 실험
- 외교, 인재 등용, 다양한 건물/정책
- 다수 AI 세력, 복잡한 승리 조건
- 멀티플레이, 클라우드 세이브, 사운드/연출 완성도
- 스토리/시나리오, 튜토리얼 정식판

## 6.3 단계별 로드맵

### 0단계 — 기술 검증 (스파이크)
- [ ] PixiJS로 모바일 브라우저에 **Hex 그리드** + 스프라이트 + **핀치-줌 카메라** 구현
- [ ] 줌 인/아웃 시 부대(토큰)→병사 표현 전환(LOD) 가능성 확인
- [ ] 화면 방향/성능 기준선 확인
- **산출물:** "Hex + 연속 줌을 이 스택으로 만들 수 있는가" 판단

### 1단계 — 전략 전투 코어
- [ ] Hex 전장, 한 타일 한 유닛, 부대 이동/선택, **전방(facing)** 전환
- [ ] 병종 3종 + 상성, **측후면 보정**, **교전 자동 계산**, 사상자
- [ ] 사기·패주 시스템, 액티브 포즈(정지·예약·재개)
- **검증:** 부대 단위 자동 전투 + 전략 개입이 그 자체로 재미있는가?

### 2단계 — 전술 확대 (★ 최우선 정체성 검증)
- [ ] 교전 확대 → 장수·개별 병종 렌더, 자동 계산 상태 인계
- [ ] 지형 연출 1~2종 + 전술 명령 2~3종
- [ ] **이중 시계**(`TIME_RATIO`) + 줌 인/아웃 흐름 + 전략 상황 알림 1~2종
- [ ] **검증: 줌 인 전술 개입이 재미있고, 시계 비율·알림이 긴장을 만드나?** ← 여기서 멈추고 판단

### 3단계 — 경영 레이어 최소 연결
- [ ] 노드 맵 + 부대 이동(턴제)
- [ ] 도시·자원·부대 편성
- [ ] 접촉 → 전투(전략/전술) → 결과 반영 루프 완성 + 자동 해결

### 4단계 — 게임 루프 완성
- [ ] AI 1세력과 한 판의 캠페인이 굴러감 (시작~승부)
- [ ] 세이브/로드, 기본 HUD
- [ ] **검증: 한 판이 끝까지 재미있는가?**

### 5단계 — 다듬기 & 확장 (선택)
- [ ] `TIME_RATIO`·알림 정책 밸런싱, 동시 다발 교전 처리 고도화
- [ ] 발동형 지형(낙석 등)·인재·외교 등 깊이 추가
- [ ] PWA/Capacitor로 앱 배포 검토

## 6.4 핵심 검증 지점 (Kill / Go)

각 단계 끝에서 냉정하게 묻는다. 재미가 없으면 다음으로 가지 말 것.

| 단계 | 질문 | 통과 못하면 |
|------|------|-------------|
| 0 | Hex + 연속 줌이 기술적으로 가능한가? | 스택/표현 방식 재검토 |
| 1 | 자동 전투 + 전략 개입이 그 자체로 재미있나? | 전략 메커니즘 재설계 |
| 2 | **줌 인 전술 개입 + 이중 시계가 긴장과 재미를 만드나?** | 정체성 재설계 (가장 중요) |
| 4 | 한 판 캠페인이 "한 판 더" 하고 싶나? | 루프/페이싱 조정 |

## 6.5 1인 개발을 위한 원칙

- **수직 슬라이스 우선** — 넓게 깔지 말고, 한 줄기를 끝까지 동작하게.
- **에셋은 임시로** — 도형·무료 에셋으로 시작. 아트는 재미 검증 후.
- **밸런싱은 나중** — 먼저 "동작", 그 다음 "재미", 마지막에 "공정".
- **줌은 단순안부터** — 옵션 C/B로 재미 검증 후 옵션 A로 끌어올린다. 처음부터 동시 실시간에 매달리지 말 것.
- **문서와 코드를 함께** — 결정이 바뀌면 이 문서들도 갱신.
