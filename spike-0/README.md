> ⚠️ **이 폴더는 검증용 실험이다.** 여기 코드를 진짜 코드베이스로 키우지 말 것.
> "된다"만 확인하고, 실제 구현은 별도로 깨끗하게 시작한다. → [doc/05 5.6](../doc/05-tech-architecture.md)

## 검증 목표 (doc/06 6.3 · doc/05 5.6.6)
연속 공간에 **덩어리(앵커+슬롯+노이즈)** 를 PixiJS로 그릴 수 있는가:
- [ ] 덩어리 렌더 + **상하 bob/애니**("꿈틀대는 덩어리")
- [ ] **facing 회전** (대열이 방향을 틀기)
- [ ] **이동 = 모임 → 도착 펼침**
- [ ] **사상자 = 좌우 끝 → 중앙 수축**
- [ ] **핀치-줌 + 팬** (모바일)
- [ ] **성능**: 스프라이트 개수 대비 FPS (모바일 상한 감)

## 조작
- **드래그** = 팬 / **휠·핀치** = 줌
- **c** = 중앙 돌격 → 두 덩어리가 붙으면 **접전**(연속 사망 팝 연출)
- **k** = 사망 10명 (개별 스프라이트 **폴짝 튀어 소멸**)
- **더블탭 / f** = ★도주 (대형 붕괴·혼란 흩어짐 후퇴)
- **r** = 리셋
- 덩어리는 자동으로 배회하며 **모임↔펼침**을 보여준다.

## 검증한 렌더 리스크
- 도주 = **대형 붕괴 흩어짐** (개별 병사 AI 없이 렌더 노이즈로)
- 사상자 = **개별 스프라이트 폴짝 튀어 소멸** — sim은 "N명 사망" 숫자만, N개 팝은 렌더
- → 덩어리 단위 연산(O(부대))인데 화면은 다발적 개별 이벤트로 보인다.

## 성능 감 잡기
`src/main.ts`의 `men`/`condense`를 바꿔 스프라이트 수를 올려가며 FPS를 본다.
(예: `condense: 4` → 스프라이트 2배). 모바일에서 몇 개까지 60fps를 유지하는지가 핵심 산출물.

## 구조 (스파이크 한정)
- `src/main.ts` — Pixi 부트스트랩·루프·입력·HUD
- `src/blob.ts` — 덩어리: 슬롯 배치·bob·facing·이동(모임/펼침)·사상자 수축
- `src/camera.ts` — 핀치줌·팬
